# Data Isolation Fix - Per-User Progress

## Проблема

Разные Firebase Auth аккаунты видели одинаковые результаты обучения и прогресс.

### Причина:
`WordsContext` использовал фиксированные ключи AsyncStorage:
- `@leet_app_known_words`
- `@leet_app_unknown_words`

Эти ключи **не содержали Firebase user UID**, поэтому:
- Пользователь A сохранял прогресс → записывал в `@leet_app_known_words`
- Пользователь B входил → читал из `@leet_app_known_words`
- Пользователь B видел прогресс пользователя A

## Решение

### Изменения:

1. **`contexts/WordsContext.tsx` - обновлен:**
   - Убрана прямая зависимость от `useAuth()` (избежание циклической зависимости)
   - Добавлены методы `loadUserData(uid)` и `clearUserData()`
   - Динамические ключи AsyncStorage с UID: `@leet_app_known_words_${uid}`
   - Данные загружаются только по явному вызову `loadUserData()`

2. **`contexts/UserDataSync.tsx` - создан:**
   - Компонент-синхронизатор между AuthContext и WordsContext
   - Отслеживает изменения `user.uid` из AuthContext
   - Автоматически вызывает `loadUserData(uid)` при входе
   - Автоматически вызывает `clearUserData()` при выходе

3. **`app/_layout.tsx` - обновлен:**
   - Добавлен `UserDataSync` в дерево провайдеров
   - Порядок: ThemeProvider → AuthProvider → WordsProvider → UserDataSync

### Архитектура:

```
AuthProvider (управляет user state)
    ↓
WordsProvider (управляет данными слов)
    ↓
UserDataSync (синхронизирует auth ↔ words)
    ↓
App (использует оба контекста)
```

## Что теперь работает

### ✅ Изоляция данных по пользователям:
- Пользователь A: данные в `@leet_app_known_words_uid_A`
- Пользователь B: данные в `@leet_app_known_words_uid_B`
- Данные не пересекаются

### ✅ Автоматическое переключение:
1. Пользователь A входит → `UserDataSync` вызывает `loadUserData(uid_A)`
2. Загружаются данные пользователя A
3. Пользователь A выходит → `UserDataSync` вызывает `clearUserData()`
4. Данные очищаются из памяти
5. Пользователь B входит → `UserDataSync` вызывает `loadUserData(uid_B)`
6. Загружаются данные пользователя B

### ✅ Персистентность:
- Каждый пользователь сохраняет прогресс в своем namespace
- При повторном входе загружается сохраненный прогресс
- Данные не теряются при переключении аккаунтов
- **Сессия сохраняется после перезапуска приложения**

### ✅ Исправлена ошибка AsyncStorage:
- Убрана циклическая зависимость между контекстами
- `WordsContext` больше не вызывает `useAuth()` напрямую
- Синхронизация происходит через отдельный компонент `UserDataSync`

## Технические детали

### Формат ключей AsyncStorage:
- **До:** `@leet_app_known_words` (глобальный)
- **После:** `@leet_app_known_words_abc123xyz` (с UID)

### Пример UID:
Firebase генерирует уникальные UID вида: `kF8xYz2mN9pQrS3tU4vW5x`

### Порядок инициализации:
1. `AuthProvider` инициализируется и слушает Firebase Auth
2. `WordsProvider` инициализируется (пустое состояние)
3. `UserDataSync` монтируется и начинает слушать `user.uid`
4. Когда `user` появляется, `UserDataSync` вызывает `loadUserData(uid)`
5. `WordsProvider` загружает данные из AsyncStorage для этого UID

### Безопасность:
- UID хранится только локально в AsyncStorage
- Данные изолированы на уровне клиента
- Для полной безопасности рекомендуется добавить Firestore с Security Rules

## Миграция старых данных

⚠️ **Важно:** Если у вас уже были данные в старых ключах (`@leet_app_known_words`), они не будут автоматически перенесены.

Если нужно сохранить старые данные:
1. Войдите в аккаунт
2. Старые данные останутся в `@leet_app_known_words`
3. Новые данные будут в `@leet_app_known_words_${uid}`
4. Можно вручную скопировать данные через AsyncStorage (если критично)

Для большинства пользователей это не проблема, так как приложение новое.
